# 01. Цели, границы и формулировка гарантии

## 1.1. Цель

StateGuard должен систематически обнаруживать и предотвращать пути, по которым обычное бизнес-
приложение может:

- принять неутверждённый внешний ввод как доменное значение;
- выполнить неразрешённый переход состояния;
- частично применить логическую команду;
- нарушить целостность при конкурирующих запросах;
- потерять или продублировать необратимый эффект;
- показать пользователю чужие, запрещённые или устаревшие данные;
- применить несовместимую migration;
- скрыть ожидаемый отказ внутри исключения или nullable-поля;
- продолжить работу после нарушения критичного предположения.

## 1.2. Формальная основа

Для состояния `s`, команды `c`, допустимой области `Safe` и реализации перехода `step` основной
safety-контракт выглядит так:

```text
Safe(s) ∧ ValidCommand(c) ∧ Assumptions
⇒
  (Accepted(step(s,c)) ∧ Safe(nextState))
  XOR
  (Rejected(step(s,c)) ∧ nextState = s)
```

Для конкурентных команд добавляется требование:

```text
Каждая committed история эквивалентна разрешённой истории модели
либо явно классифицированному уровню eventual consistency.
```

Для наблюдений:

```text
response ∈ Projection(VisibleState(databaseState, actor))
```

Для внешних эффектов:

```text
CommittedDomainEvent
⇒ eventually(EffectAppliedExactlyAccordingToDeclaredDeliverySemantics)
```

Последняя формула не означает абстрактное exactly-once. Реализация обычно использует at-least-once,
idempotency key, deduplication и reconciliation.

## 1.3. Что означает зелёный verdict

`GREEN BY EVIDENCE` допустим только при одновременном выполнении условий:

- текущий repository manifest полностью просканирован;
- все review units относятся к текущим SHA-256;
- каждая critical/high поверхность имеет владельца и evidence;
- critical/high invariants закрыты утверждённым классом доказательства;
- открытых critical/high findings нет;
- model checker не нашёл counterexample в заявленной области;
- migrations и real-database tests прошли;
- AI-only evidence не используется вместо обязательного formal/static/test evidence;
- assumptions и исключения опубликованы;
- toolchain versions и ruleset hashes сохранены.

## 1.4. Что StateGuard не обещает

StateGuard не заявляет:

- отсутствие дефектов в неописанных требованиях;
- корректность оборудования, компилятора и ОС вне принятой доверенной базы;
- полноту любого отдельного статического анализатора;
- доказательство произвольного динамического кода без ограничений;
- корректность внешнего сервиса, нарушающего заявленный контракт;
- liveness без fairness и инфраструктурных assumptions;
- безопасность, если команда отключила constraints, rules или проверки.

Эти ограничения фиксируются позитивно: StateGuard гарантирует свойства внутри явно утверждённого
scope и доверенной базы.

## 1.5. Классы evidence

Каждый proof obligation получает один или несколько классов evidence:

| Код | Значение |
|---|---|
| `proved-model` | Rodin/Event-B proof obligation закрыт |
| `checked-model` | ProB не нашёл violation в полностью/ограниченно исследованной области |
| `proved-smt` | Z3 подтвердил узкую формулу |
| `proved-static` | детерминированный графовый/статический запрос подтвердил свойство |
| `enforced-db` | PostgreSQL constraint/index/RLS/trigger физически запрещает состояние |
| `verified-db` | свойство проверено на настоящей БД |
| `verified-test` | воспроизводимый integration/concurrency test прошёл |
| `reviewed-human` | инженер подтвердил смысл и evidence |
| `reviewed-ai` | локальный агент дал обоснованное заключение |
| `waived` | утверждённый риск с владельцем и сроком |

Правило доверия:

```text
reviewed-ai ≤ reviewed-human < verified/proved evidence
```

ИИ помогает сформулировать гипотезу и найти путь. Он не повышает finding до доказанного без
воспроизводимого evidence.

## 1.6. Severity

- `critical`: нарушение денег, прав доступа, необратимых операций, массовая порча/утечка данных.
- `high`: достижимое нарушение ключевого бизнес-инварианта или серьёзная конкурентная гонка.
- `medium`: ограниченное неправильное поведение, отказ или неполнота защиты.
- `low`: локальный дефект с малым воздействием.
- `info`: evidence gap, hotspot или рекомендация, не являющаяся доказанным дефектом.

Severity не равна confidence. Дефект может быть critical с низкой уверенностью; тогда он требует
срочного подтверждения, а не немедленного обвинительного verdict.
