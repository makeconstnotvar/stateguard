# 03. Контракт репозитория

## 3.1. Обязательная структура

```text
repo/
├── .stateguard/
│   ├── stateguard.yaml
│   ├── specification.yaml
│   ├── mappings.yaml
│   ├── rules/
│   ├── audit.db
│   ├── results/
│   ├── reports/
│   ├── work/
│   └── .gitignore
├── sonar-project.properties
└── ... application code
```

В VCS обычно хранятся:

- `stateguard.yaml`;
- specification;
- mappings;
- project-specific rules;
- approved waivers в декларативной форме;
- model files;
- generated test fixtures, если они стабильны.

Не хранятся:

- `audit.db` и WAL files;
- raw CPG;
- transient logs;
- tokens;
- AI prompt cache с чувствительными excerpts.

## 3.2. `stateguard.yaml`

Конфигурация определяет только механический scope:

- project identity;
- source roots и excludes;
- generated paths;
- critical paths;
- расположение specification/mappings;
- analyzer outputs;
- Sonar project key.

Бизнес-инварианты сюда не помещаются. Иначе operational config превращается в неуправляемую
спецификацию.

## 3.3. `specification.yaml`

Это технологически нейтральная модель:

- entities;
- state domains;
- invariants;
- commands;
- guards;
- outcomes;
- observations;
- effects;
- assumptions.

Она должна проходить JSON Schema validation и semantic validation:

- все references разрешаются;
- IDs уникальны;
- initial state принадлежит values;
- outcome IDs уникальны;
- каждый critical invariant имеет enforcement policy;
- command сохраняет перечисленные invariants;
- effect с irreversible semantics имеет idempotency/unknown-outcome policy.

## 3.4. `mappings.yaml`

Mapping связывает технологически нейтральную модель с реализацией:

```text
CMD-ORDER-SUBMIT
  → HTTP route
  → decoder
  → authorization function
  → command handler
  → SQL statement
  → UI action
  → integration/concurrency tests
```

Mapping не должен зависеть только от путей. Предпочтительный selector hierarchy:

1. stable symbol ID из analyzer;
2. fully qualified symbol;
3. route/query/constraint name;
4. path + symbol;
5. path glob как fallback.

## 3.5. Stable IDs

ID не переименовывается вместе с маркетинговым названием. Используются:

```text
ENT-ORDER
STATE-ORDER
INV-ORDER-001
CMD-ORDER-SUBMIT
OUT-ORDER-SUBMIT-STALE
OBS-ORDER-DETAIL
EFFECT-PAYMENT-REQUEST
PO-CMD-ORDER-SUBMIT-AUTH-DOMINANCE
```

Описание можно менять, ID остаётся ссылочным контрактом истории.

## 3.6. Generated artifacts

Generated code не исключается безусловно. Он классифицируется:

- generated source, который компилируется и исполняется;
- generated bundle;
- vendored dependency;
- generated schema/client;
- migration generated by tool.

StateGuard может не проводить ручной review generated source, но обязан:

- хранить hash генератора;
- хранить hash input schema;
- анализировать generated output механическими инструментами, если он исполняется;
- проверять воспроизводимость generation;
- инвалидировать evidence при изменении генератора.

## 3.7. Monorepo

Для monorepo есть root policy и project overlays:

```text
/.stateguard/policy.yaml
/apps/order/.stateguard/stateguard.yaml
/apps/billing/.stateguard/stateguard.yaml
/packages/shared/.stateguard/component.yaml
```

Manifest строится один раз, затем компоненты получают собственные review units. Shared package
изменяет доказательства всех зависимых компонентов через dependency graph.
